# Track Specification - Comprehensive System Refactoring & Stability Improvement

## Overview
本プロジェクトのバックエンド（FastAPI）およびフロントエンド（Next.js）全体を対象とし、コードの可読性、実行パフォーマンス、テストの容易性、およびモジュール性を向上させるための包括的なリファクタリングを行います。最近導入した「事前計算データ」や「緩和された認証ポリシー」をシステム全体に馴染ませ、より堅牢な基盤を構築します。

## Objectives
- **Readability**: 複雑化したシミュレーションロジックやUIコンポーネントの構造を整理し、意図が伝わりやすいコードにする。
- **Performance**: 重複する計算や不要なレンダリングを削減し、システム全体の応答性を高める。
- **Testability/Stability**: モジュール間の結合度を下げ、ユニットテストや結合テストをより確実に行えるようにする。
- **DRY/Modularization**: 共通化可能なロジック（認証、データフェッチ、シミュレーション部品）を共通モジュールに切り出す。

## Functional Requirements (Refactoring Focus)
### Backend
- `app/simulation.py`: 数学的な安定性を維持しつつ、計算ロジックをよりモジュール化し、単体テストをさらに細かく行えるように整理。
- `app/main.py`: エンドポイントの定義を整理し、共通の依存関係（Depends）やエラーハンドリングを標準化。
- `app/seed_assets.py`: ハードコードされたデータとスクリプト本体を分離し、将来的な資産追加を容易にする。
- 認証周り: `get_optional_user_id` と `get_current_user_id` の使い分けを徹底し、コードの重複を排除。

### Frontend
- コンポーネントの分割: 肥大化したページコンポーネント（特にシミュレーション画面）を、小さな責務を持つサブコンポーネントに分割。
- カスタムフックの標準化: `useEfficientFrontier` 等の React Query フックのパターンを統一し、エラーハンドリング一元化。
- 状態管理: `zustand` のストア設計を見直し、`hasHydrated` フラグの利用状況をプロジェクト全体で一貫させる。
- 型定義の整理: `types/simulation.ts` 等の型定義を整理し、バックエンドのスキーマとの同期をより確実に。

## Non-Functional Requirements
- **完全な後方互換性**: 既存のAPIインターフェースを変更せず、リファクタリング前後で機能的な劣化（リグレッション）が発生しないこと。
- **テストカバレッジの維持/向上**: 重要なロジックに対して追加のテストケースを記述する。

## Acceptance Criteria
- 全てのバックエンドテスト（`pytest`）がパスすること。
- 全てのフロントエンドテスト（`vitest`）がパスすること。
- コードの重複が削減され、プロジェクト構造がより直感的になっていること。
- 以前発生した「グラフの数値爆発」のような問題に対するガードレールがコードレベルで整備されていること。

## Out of Scope
- 大規模な新規機能の追加（リファクタリングに伴う軽微な改善は含む）。
- データベーススキーマの破壊的な変更。
