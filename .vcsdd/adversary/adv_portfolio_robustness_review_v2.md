## 1. Critical Flaws Found

### đźš¨ [CRITICAL] Portfolio Creation is Broken (Regression)
In `backend/app/crud.py`, the `create_portfolio` function attempts to create `PortfolioAllocation` records before the `Portfolio` record has been flushed to the database.
- **Root Cause**: `db_portfolio.id` is `None` until the session is flushed or committed because the ID is generated by the database default (or Python-side default that triggers on flush).
- **Symptom**: `IntegrityError: NOT NULL constraint failed: portfolio_allocations.portfolio_id`.
- **Impact**: Users cannot create any portfolio that has allocations (the primary use case).

### đźš¨ [MAJOR] Duplicate Asset Codes Allowed
SPEC-012 explicitly states: *"Duplicate Asset Codes: A portfolio should not have duplicate asset entries."*
- **Issue**: The current implementation in `create_portfolio` and `update_portfolio` does not check for duplicate `asset_code` in the input list. It will happily create multiple allocations for the same asset.
- **Impact**: Calculation errors in simulation and confusing UI.

### âš ď¸Ź [MEDIUM] Missing Referential Integrity (Simulation Results)
SPEC-012 states: *"Deleting a portfolio must safely handle (cascade or clear) linked simulation results."*
- **Issue**: `SimulationResult` has a foreign key to `portfolios.id`, but the relationship is not defined with `cascade` in the `Portfolio` model, nor is the FK defined with `ondelete="CASCADE"` or `ondelete="SET NULL"` in the DB schema.
- **Impact**: Deleting a portfolio will fail with a `ForeignKeyViolation` (Postgres) or leave orphaned results (SQLite).

### âš ď¸Ź [MEDIUM] Potential 500 Errors in Simulation Results
While many routes now have `try...except` blocks, the `/api/simulation-results` POST endpoint and several `GET` endpoints (e.g., `/api/assets`, `/api/market-summary`) do **not** have local error handling.
- **Issue**: Saving a simulation result that contains `NaN` will crash on Postgres (JSONB) with a 500 error because it hits the global middleware instead of returning a 400.
- **Impact**: System instability and poor error reporting for simulation failures.

## 2. Race Conditions & Data Integrity

- **Asset Verification Race**: Verification of asset existence happens within the transaction, which is good. However, because there is no Foreign Key from `portfolio_allocations.asset_code` to `asset_data.asset_code`, an asset could still be deleted between the check and the commit, resulting in an orphan allocation.
- **Transaction Atomicity**: `update_portfolio` correctly performs `delete()` and `add()` within a single transaction. However, without explicit row locking (`with_for_update()`), concurrent updates to the same portfolio could lead to race conditions where allocations from two different requests are interleaved or duplicated.

---

## 3. Precision & Numerical Stability

- **Suboptimal Conversions**: In `main.py`, `read_portfolios` performs a round-trip conversion: `Decimal -> float -> str -> Decimal`.
  ```python
  a.weight = Decimal(str(np.nan_to_num(float(a.weight))))
  ```
  While likely lossless for 6 decimal places, this is idiomatic "code smell" and introduces unnecessary risk and overhead. Numerical stability should be handled at the source (simulation output) or via a custom JSON encoder, not by manual casting during retrieval.

---

## 4. Recommendations for Remediation

1.  **Fix `create_portfolio`**: Call `db.flush()` immediately after `db.add(db_portfolio)` to populate the `id` field before creating allocations.
2.  **Enforce Uniqueness**: Add a check in `schemas.py` or `crud.py` to ensure `asset_code` is unique within the allocations list.
3.  **Define Cascades**: Update `models.py` to include `ondelete="CASCADE"` on the `SimulationResult.portfolio_id` foreign key or add a relationship with `cascade="all, delete-orphan"` in the `Portfolio` model.
4.  **Harden Schema**: Add an explicit `ForeignKey` constraint from `PortfolioAllocation.asset_code` to `AssetData.asset_code`.
5.  **Standardize Error Blocks**: Wrap the remaining endpoints in `try...except Exception` blocks to ensure they return 400/422 instead of triggering the 500 middleware.
